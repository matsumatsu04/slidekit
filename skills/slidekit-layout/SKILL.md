# slidekit-layout — 構図パターン抽出・定義

## このスキルは何をするか

1 枚のスライドの **構図（要素の置き方）だけ**を、再利用可能なパターンとして
`SLIDEKIT-LAYOUT-{name}.md` に書き出す。色・フォント・実コンテンツは**一切含めない**（それらはデザインシステムと設計書の役割）。
あわせて、構図が分かる **グレースケールの確認用HTML**を生成する（色を抜くことで「構図そのもの」に集中できる）。

成果物。
- `patterns/{category}/SLIDEKIT-LAYOUT-{name}.md`
- 構図確認用のグレースケール HTML（個別 or まとめプレビュー）

形式の正は常に `SPEC.md` の「2. SLIDEKIT-LAYOUT-{name}.md」セクション。**着手前に必ず `SPEC.md` を読む。**

> なぜ色を持たないのか：構図とデザインを分離することで、1 つの構図を「どんなデザインシステムとも」
> 組み合わせて再利用できる。これが SlideKit の核。

## 2つのモード

### A. 抽出モード（キャプチャから）
ユーザーが「この構図いいな」というスライド画像を渡してくる場合。
1. 画像から **要素の配置だけ**を読み取る（見出しの位置、本文ブロック、図の位置、カラム数、余白配分）。
2. 配置を canvas に対する **% の領域**に変換する（色・文字内容は無視する）。
3. `SLIDEKIT-LAYOUT-{name}.md` に落とす。

### B. 設計モード（ゼロから）
「2カラム比較の構図がほしい」など、目的から新規に設計する場合。
1. category（cover/agenda/section/content/list/comparison/data/process/quote/closing）を決める。
2. 要素と配置を設計し、% 領域で定義する。

## 手順

### 1. 入力と category を確認する
画像があれば抽出モード、なければ設計モード。`category` を 10 種から決める。

### 2. 構造を % で定義する
- 各領域を `{ x, y, w, h }`（canvas に対する %）で記述する。
- 余白・カラム間（gutter）も配分に含める。
- 要素数の目安（例：カードは 3 つまで、箇条は 5 行まで）を決める。
- **色・フォント・実際の文言は書かない。**

### 3. SLIDEKIT-LAYOUT-{name}.md を書く
`SPEC.md` の形式で `Meta` / `Structure(YAML)` / `Slots` / `Authoring Notes` を埋める。
- `pattern` は内容が分かる kebab-case（例：`compare-two-col`, `kpi-row`, `process-steps`）。
- `Slots` には各領域の役割と推奨文字数/個数を書く。
- `Authoring Notes` には、AI に再現させるときの注意（崩れ方の回避、要素上限など）を書く。

### 4. グレースケール確認HTMLを作る
構図が一目で分かるよう、**グレー階調だけ**で領域を四角で示す HTML を作る。
- 出力先は `patterns/{category}/{name}.preview.html`（個別）に固定する。
  ライブラリ全体の一覧は `patterns/preview.html`（まとめ）に追記する。
- 各領域に役割ラベル（"見出し" "本文" "図" など）を入れる。
- 16:9 の枠で表示する。
- 色は使わない（#fff / #eee / #ddd / #999 / #333 程度のグレーのみ）。

### 5. 出力して確認を促す
パスを伝え、グレースケール確認HTMLで構図を確認してもらう。

## 完了基準（Definition of Done）
- `SLIDEKIT-LAYOUT-{name}.md` が `SPEC.md` の全セクションを満たす。
- `Structure` が % 領域で、合計が破綻していない（重なり・はみ出しがない）。
- 色・フォント・実コンテンツが混入していない。
- グレースケール確認HTMLで構図が再現できている。

## 自己検証
出力前に確認し、結果は**ユーザーへの最終報告に記載**する（成果物ファイル本体には書かない）。
1. 色・フォント・実コンテンツが入っていないか（構図だけか）
2. 各 region が `x + w <= 100` / `y + h <= 100` を満たすか（はみ出しなし）
3. 意図しない region 同士の重なりがないか
4. `Slots` のスロット名が `Structure.regions[].id` と一致しているか
5. DESIGN の Frame 予約領域（下端 8% 程度）を避けているか
6. category が 10 種のいずれかか

## やってはいけないこと
- 色・フォントを定義しない（→ `slidekit-design`）。
- 実際のプレゼン文言を入れない（→ `slidekit-assemble`）。
- 既存の他者リポジトリのファイルを転記・改変して流用しない。構図の“種類”は一般的だが、
  各パターンは本スキルの形式で**自分で書き起こす**こと。
