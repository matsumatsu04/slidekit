# slidekit-layout — 構図パターン抽出・定義

## このスキルは何をするか

1 枚のスライドの **構図（コンテンツエリアの要素の置き方）だけ**を、再利用可能なパターンとして
`patterns/SLIDE-PATTERN-{name}/` に書き出す。色・フォント・実コンテンツ、タイトル枠・ページ番号・装飾は
**一切含めない**（それらは DESIGN／DECK の役割）。あわせて、構図が分かる **グレースケールの確認用HTML**を生成する。

成果物（1 パターン ＝ 1 ディレクトリ）。
- `patterns/SLIDE-PATTERN-{name}/SLIDE-PATTERN-{name}.md`（構図定義）
- `patterns/SLIDE-PATTERN-{name}/SLIDE-PATTERN-{name}.html`（グレースケール確認用・960×540）

形式の正は常に `SPEC.md` の「2. SLIDE-PATTERN-{name}.md」セクション。**着手前に必ず `SPEC.md` と既存パターン数件を読む**
（既存ライブラリと体裁・粒度を揃えるため）。

> なぜ色を持たないのか：構図とデザインを分離することで、1 つの構図を「どんなデザインシステムとも」
> 組み合わせて再利用できる。これが SlideKit の核。

## 2つのモード

### A. 抽出モード（キャプチャから）
ユーザーが「この構図いいな」というスライド画像を渡してくる場合。
1. 画像から **要素の配置だけ**を読み取る（見出しの位置、本文ブロック、図の位置、カラム数、余白配分）。
2. 構造として言語化する（色・文字内容は無視する）。

### B. 設計モード（ゼロから）
「2カラム比較の構図がほしい」など、目的から新規に設計する。

## 手順

### 1. 入力と category を確認する
画像があれば抽出モード、なければ設計モード。`category` を 14 種から決める：
`表紙 / セクション / 目次 / コンテンツ（本文） / リスト / ステップ / 図解・ダイアグラム /
カード / グラフ / テーブル / KPI / まとめ / FAQ / プロフィール`。
※ 既存に似たパターンがないか `patterns/SLIDE-PATTERN-INDEX.md` を確認し、重複を避ける。

### 2. SLIDE-PATTERN-{name}.md を書く
`SPEC.md` の形式（既存ライブラリと同じ）で書く。
- 前書き：「コンテンツエリアのレイアウト定義であり、タイトル枠・ページ番号・装飾は DESIGN 側で定義する」旨。
- `## Overview`：パターン名 / 概要 / 適したシーン
- `## Structure（構造）`：カラム数・行・要素配置を YAML 風に（色・実値は書かない）
- `## Elements（各要素の役割）`：要素 / 配置 / 役割の表
- `## Usage Guide（AIへの使い方）`：プロンプト例と注意点
- `name` は内容が分かる kebab-case（既存と命名トーンを揃える）。

### 3. グレースケール確認HTMLを作る
構図が一目で分かるよう、**グレー階調だけ**の 960×540 スライドHTMLを作る（既存パターンの `.html` に体裁を合わせる）。
- 各エリアに役割ラベルを入れる。色は使わない（#fff / #eee / #ccc / #999 / #333 程度）。

### 4. INDEX と manifest を更新する（必須）
- `patterns/SLIDE-PATTERN-INDEX.md` の該当カテゴリ末尾に行（パターン名 / 概要 / 適したシーン）を追記。
- `patterns/manifest.json` に `{name, category, summary}` を追記（ギャラリーが参照）。

### 5. 出力して確認を促す
パス（.md / .html）を伝え、グレースケール確認HTMLで構図を確認してもらう。ギャラリーにも自動で並ぶ旨を添える。

## 完了基準（Definition of Done）
- `SLIDE-PATTERN-{name}.md` が `SPEC.md` の全セクション（Overview/Structure/Elements/Usage Guide）を満たす。
- 色・フォント・実コンテンツ、タイトル枠・ページ番号が混入していない。
- グレースケール `.html` で構図が再現できている。
- INDEX と manifest.json に追記済み（カテゴリは 14 種のいずれか）。

## 自己検証
出力前に確認し、結果は**ユーザーへの最終報告に記載**する（成果物ファイル本体には書かない）。
1. 色・フォント・実コンテンツ・タイトル枠が入っていないか（コンテンツエリアの構図だけか）
2. `Overview / Structure / Elements / Usage Guide` が揃っているか
3. category が 14 種のいずれかか
4. INDEX と manifest.json を更新したか
5. 既存ライブラリと命名・粒度・体裁が揃っているか

## やってはいけないこと
- 色・フォントを定義しない（→ `slidekit-design`）。
- 実際のプレゼン文言・タイトル枠・ページ番号を入れない（→ DESIGN／`slidekit-assemble`）。
- INDEX・manifest の更新を忘れない（忘れるとギャラリーや assemble から見えない）。
